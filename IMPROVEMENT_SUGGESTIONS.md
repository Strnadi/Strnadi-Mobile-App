# Recommended Improvements for Strnadi Mobile App

Based on a review of the codebase, the following suggestions are proposed to improve architecture, maintainability, security, and testing.

## 1. Architectural Refactoring

### Issue: The "God Class" Anti-Pattern
The class `DatabaseNew` (in `lib/database/databaseNew.dart`) currently handles too many responsibilities:
- Database operations (CRUD)
- API networking (HTTP/Dio calls)
- File system management
- Business logic

This violates the Single Responsibility Principle, making the code difficult to test and maintain.

### Recommendation: Layered Architecture
Refactor the application into distinct layers:

1.  **Service Layer (API Client):** Extract all HTTP/Dio logic into a dedicated service.
2.  **Repository Layer:** Abstract data sources. The repository decides whether to get data from the local database or the remote API and coordinates synchronization.
3.  **State Management:** Move business logic out of UI widgets.

#### Example Refactoring

**Step 1: Extract API Client**
Create `lib/services/api_client.dart` to handle all networking.

```dart
// lib/services/api_client.dart
import 'package:dio/dio.dart';
import 'package:strnadi/config/config.dart';

class ApiClient {
  final Dio _dio;

  ApiClient({Dio? dio}) : _dio = dio ?? Dio(BaseOptions(baseUrl: 'https://${Config.host}'));

  Future<Response> postRecording(Map<String, dynamic> data, String token) {
    return _dio.post(
      '/recordings',
      data: data,
      options: Options(headers: {'Authorization': 'Bearer $token'}),
    );
  }
}
```

## 2. Code Quality & Maintenance

### A. Fix Hardcoded API Paths & Strings
**Issue:** API paths (e.g., `/recordings/part-new`) and user-facing strings are hardcoded in logic files.
**Fix:**
-   Move API endpoints to a `ApiConstants` class.
-   Ensure **all** user-facing strings use the `localization/localization.dart` system. (e.g., in `streamRec.dart`).

### B. Improve Database Security
**Issue:** Use of `rawQuery` with string interpolation creates potential for errors and SQL injection.
**Fix:** Always use parameterized queries (`whereArgs`).

*Current (Risky):*
```dart
await db.rawQuery("SELECT * FROM recordingParts WHERE RecordingId = $id");
```

*Recommended (Safe):*
```dart
await db.query("recordingParts", where: "RecordingId = ?", whereArgs: [id]);
```

### C. Null Safety
**Issue:** Usage of `late` variables (e.g., `_audioRecorder` in `streamRec.dart`) initialized in methods like `onStart` can cause runtime crashes if the lifecycle isn't perfect.
**Fix:** Prefer nullable types (`AudioRecorder?`) and check for nulls before usage.

## 3. State Management

**Issue:** `LiveRec` (in `streamRec.dart`) manages permissions, audio state, file I/O, and UI updates, resulting in a complex and brittle widget.
**Recommendation:** Adopt a state management solution (like **Riverpod**, **Bloc**, or **Provider**) to separate logic from the UI. This allows for testing the recording logic without the UI.

## 4. Testing (Critical Gap)

**Issue:** The project lacks comprehensive testing.
**Recommendation:**
1.  **Unit Tests:** specific tests for the new `ApiClient` and `Repository` classes using `mockito`.
2.  **Widget Tests:** Basic smoke tests for widgets to ensure they render and handle tap events correctly.

## 5. Dependency Management

**Issue:** `pubspec.yaml` contains loose version constraints (`any`).
**Fix:** Pin dependencies to specific versions (e.g., `sqflite: ^2.0.0`) to prevent breaking changes from auto-updates.
